2.8 ИНФОРМАЦИОННАЯ СТРУКТУРА ПРИЛОЖЕНИЯ

2.8.1 Общая схема пользовательского интерфейса веб-приложения

Интерфейс мессенджера построен по модульной архитектуре с использованием компонентного подхода. Основные элементы интерфейса включают:

- Главная панель навигации с аватаром пользователя, статусом онлайн/оффлайн, меню настроек и уведомлениями
- Боковая панель с чатами, содержащая список диалогов с индикаторами непрочитанных сообщений, статусами доставки и временными метками последнего сообщения
- Центральная область чата с историей сообщений, отображением статусов доставки и прочтения, возможностью вложения медиафайлов
- Панель ввода сообщения с кнопками прикрепления файлов, emoji-picker и отправки
- Боковая панель профиля пользователя и информации о чате (участники, настройки безопасности)

Интерфейс полностью адаптирован для работы на различных устройствах - от десктопных компьютеров до планшетов и смартфонов. Реализована система тем оформления (светлая, темная, контрастная) с возможностью персональной настройки цветовой схемы.

2.8.2 Перечень и иерархия уникальных экранных форм веб-приложения

1. Форма авторизации/регистрации
   - Вход по логину/паролю
   - Вход через социальные сети
   - Регистрация нового пользователя
   - Восстановление пароля

2. Главная страница (чат-лист)
   - Список чатов
   - Поиск по чатам
   - Создание нового чата
   - Фильтрация по типам чатов

3. Экран чата
   - История сообщений
   - Панель отправки сообщений
   - Прикрепленные файлы
   - Участники чата

4. Профиль пользователя
   - Редактирование данных
   - Настройки безопасности
   - Привязка устройств
   - История активности

5. Настройки приложения
   - Тема оформления
   - Уведомления
   - Конфиденциальность
   - Безопасность

6. Административная панель
   - Управление пользователями
   - Модерация контента
   - Статистика системы
   - Журнал событий

2.8.3 Архитектура взаимодействия компонентов мессенджера

Система построена по модульной монолитной архитектуре с возможностью последующего разделения на микросервисы. Архитектурные компоненты:

1. Клиентская часть:
   - Веб-интерфейс (React.js)
   - Мобильное приложение (на базе React Native)
   - WebSocket-клиент для реального времени
   - Локальное хранилище данных

2. Серверная часть:
   - REST API Gateway
   - WebSocket-сервер
   - Сервис аутентификации (JWT)
   - Сервис управления чатами
   - Сервис хранения и обработки сообщений
   - Сервис уведомлений
   - Сервис хранения файлов
   - Сервис шифрования

3. Внешние сервисы:
   - PostgreSQL (основная СУБД)
   - Redis (кэширование и сессии)
   - MinIO (объектное хранилище)
   - SMTP-сервер (email-уведомления)

Все компоненты взаимодействуют через стандартизированные API с использованием JSON для обмена данными. Для реального времени используется WebSocket с протоколом STOMP.

2.8.4 Структура базы данных и сущности мессенджера

Основные сущности базы данных:

1. Пользователь (User):
   - id (BIGSERIAL PRIMARY KEY)
   - username (VARCHAR(50) UNIQUE NOT NULL)
   - email (VARCHAR(100) UNIQUE NOT NULL)
   - password_hash (VARCHAR(255) NOT NULL)
   - first_name (VARCHAR(50))
   - last_name (VARCHAR(50))
   - avatar_url (VARCHAR(255))
   - role (UserRole ENUM DEFAULT 'USER')
   - status (UserStatus ENUM DEFAULT 'ACTIVE')
   - created_at (TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP)
   - updated_at (TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP)

2. Чат (Chat):
   - id (BIGSERIAL PRIMARY KEY)
   - name (VARCHAR(100))
   - description (TEXT)
   - type (ChatType ENUM NOT NULL DEFAULT 'PRIVATE')
   - encrypted (BOOLEAN DEFAULT FALSE)
   - encryption_key (VARCHAR(255))
   - encryption_algorithm (VARCHAR(50) DEFAULT 'AES')
   - security_level (VARCHAR(20) DEFAULT 'LIMITED')
   - avatar_url (VARCHAR(255))
   - created_by (BIGINT REFERENCES users(id))
   - created_at (TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP)
   - updated_at (TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP)

3. Участник чата (UserChat):
   - id (BIGSERIAL PRIMARY KEY)
   - chat_id (BIGINT NOT NULL REFERENCES chats(id) ON DELETE CASCADE)
   - user_id (BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE)
   - role (ChatRole ENUM DEFAULT 'MEMBER')
   - joined_at (TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP)
   - left_at (TIMESTAMP WITH TIME ZONE)
   - UNIQUE(chat_id, user_id)

4. Сообщение (Message):
   - id (BIGSERIAL PRIMARY KEY)
   - chat_id (BIGINT NOT NULL REFERENCES chats(id) ON DELETE CASCADE)
   - sender_id (BIGINT NOT NULL REFERENCES users(id))
   - content (TEXT)
   - message_type (MessageType ENUM DEFAULT 'TEXT')
   - reply_to_message_id (BIGINT REFERENCES messages(id))
   - status (MessageStatus ENUM DEFAULT 'SENT')
   - created_at (TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP)
   - updated_at (TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP)

5. Файл (File):
   - id (BIGSERIAL PRIMARY KEY)
   - original_name (VARCHAR(255) NOT NULL)
   - stored_name (VARCHAR(255) NOT NULL)
   - content_type (VARCHAR(100))
   - size (BIGINT NOT NULL)
   - bucket_name (VARCHAR(100) NOT NULL)
   - object_key (VARCHAR(500) NOT NULL)
   - uploaded_by (BIGINT REFERENCES users(id))
   - uploaded_at (TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP)
   - expires_at (TIMESTAMP WITH TIME ZONE)

6. Связь сообщения и файлов (MessageFile):
   - id (BIGSERIAL PRIMARY KEY)
   - message_id (BIGINT NOT NULL REFERENCES messages(id) ON DELETE CASCADE)
   - file_id (BIGINT NOT NULL REFERENCES files(id) ON DELETE CASCADE)
   - UNIQUE(message_id, file_id)

Таблицы снабжены необходимыми индексами для оптимизации производительности:
- idx_users_username, idx_users_email, idx_users_role, idx_users_status
- idx_chats_type, idx_chats_created_by
- idx_chat_members_chat_id, idx_chat_members_user_id, idx_chat_members_role
- idx_messages_chat_id, idx_messages_sender_id, idx_messages_created_at, idx_messages_reply_to
- idx_files_uploaded_by, idx_files_uploaded_at
- idx_message_files_message_id, idx_message_files_file_id

Для оптимизации производительности при работе с большими объемами сообщений реализована система партиционирования таблицы сообщений по временным интервалам (месяцы).

2.8.5 Форматы обмена данными и API endpoints

REST API:

1. Аутентификация:
   - POST /api/auth/register - регистрация пользователя
   - POST /api/auth/login - вход в систему
   - POST /api/auth/refresh - обновление токена
   - GET /api/auth/profile - получение профиля

2. Управление пользователями:
   - GET /api/users/me - получить текущего пользователя
   - PUT /api/users/me - обновить профиль
   - GET /api/users/search?q={query} - поиск пользователей

3. Управление чатами:
   - GET /api/chats - получить чаты пользователя
   - POST /api/chats - создать чат
   - GET /api/chats/{chatId} - получить чат по ID
   - PUT /api/chats/{chatId}/type - изменить тип чата
   - PUT /api/chats/{chatId}/encryption - настроить шифрование
   - PUT /api/chats/{chatId}/moderator - назначить модератора
   - DELETE /api/chats/{chatId}/members/{userId} - удалить участника

4. Управление участниками чата:
   - POST /api/chats/{chatId}/members?userId={userId}&role={role} - добавить участника
   - DELETE /api/chats/{chatId}/members/{userId} - удалить участника
   - PUT /api/chats/{chatId}/members/{userId}/role - изменить роль участника

5. Работа с сообщениями:
   - GET /api/messages/{chatId}?page={page}&size={size} - получить сообщения чата
   - POST /api/messages - отправить сообщение
   - PUT /api/messages/{messageId}/read - отметить сообщение как прочитанное
   - DELETE /api/messages/{messageId} - удалить сообщение

6. Работа с файлами:
   - POST /api/files/upload - загрузить файл
   - GET /api/files/{fileId} - получить файл
   - DELETE /api/files/{fileId} - удалить файл

WebSocket API:

1. Подключение:
   - /ws - точка подключения WebSocket
   - /app/chat.send - отправить сообщение
   - /app/chat.join - присоединиться к чату
   - /app/chat.leave - покинуть чат
   - /user/queue/messages - приватные сообщения пользователя
   - /topic/chat.{chatId} - широковещательные сообщения чата

2.8.6 Архитектура безопасности и шифрования

Система безопасности реализована на нескольких уровнях:

1. На уровне сети:
   - Все соединения осуществляются по протоколу HTTPS (TLS 1.3+)
   - Настройка HSTS (HTTP Strict Transport Security) для принудительного использования HTTPS
   - Защита от DDoS-атак через Nginx и cloudflare CDN
   - Ограничение скорости запросов (rate limiting) с автоматической блокировкой подозрительных IP-адресов

2. На уровне приложения:
   - Аутентификация через JWT-токены с двухфакторной аутентификацией (опционально)
   - Хранение паролей с использованием алгоритма bcrypt с высоким коэффициентом сложности
   - Применение принципа минимальных привилегий для всех операций
   - Защита от распространенных веб-уязвимостей:
     * SQL-инъекции: параметризованные запросы через Spring Data JPA
     * XSS: автоматическое экранирование вывода в HTML
     * CSRF: токены для всех изменяющих состояние запросов
     * Clickjacking: соответствующие заголовки X-Frame-Options
   - Валидация и санитизация всех входных данных
   - Аудит всех критических операций (изменение настроек, удаление данных, вход в систему)

3. На уровне данных:
   - Шифрование чувствительных данных при хранении (AES-256)
   - Разделение прав доступа к данным на уровне базы данных
   - Шифрование данных при передаче между компонентами
   - Использование отдельных баз данных для разных типов информации
   - Автоматическое удаление персональных данных при удалении аккаунта

4. Дополнительная защита контента:
   - Сквозное (end-to-end) шифрование сообщений с использованием Signal Protocol
   - Визуальная индикация уровня шифрования в интерфейсе (зеленый - защищенный, желтый - частично защищенный, красный - небезопасный)
   - Автоматическое обнаружение потенциально опасного контента
   - Модерация контента с возможностью административного контроля

Система соответствует требованиям законодательства:
   - Реализовано GDPR/152-ФЗ для персональных данных
   - Поддержка экспорта и удаления данных пользователя по запросу
   - Прозрачная политика конфиденциальности для пользователей
   - Регулярное проведение аудитов безопасности и пентестов
